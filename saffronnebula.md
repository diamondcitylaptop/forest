# Olive Link Navigator

Olive Link Navigator 是一个面向开发者与技术研究人员的轻量化外链资源导航与结构化文档聚合系统。该项目定位于解决个人或团队在查阅分散于多个仓库、多个平台的技术文档、笔记、参考链接时面临的检索效率低、结构混乱、缺乏统一入口的问题。Olive Link Navigator 本身不存储大量文档实体，而是通过约定化的 Markdown 索引文件，将来自不同来源的高质量外链、技术备忘录、架构说明、部署手册等资源进行集中登记、分类与交叉引用，从而构建一个可维护、可扩展、可审计的技术外链知识图谱。目标用户包括需要维护个人技术知识库的开发者、需要为团队搭建内部技术文档导航的架构师，以及需要将分散的调研笔记进行系统化整理的研发管理者。

## 功能概览

- **结构化外链索引**：支持以 Markdown 文件为记录单元，每个文件内可包含标题、标签、摘要、原始链接、关联项目、最后验证时间等元数据，便于后续自动化扫描与展示。

- **分类标签体系**：内置轻量级标签系统，允许为每条外链标记多个分类标签，例如“部署指南”、“性能调优”、“安全加固”、“API 参考”、“故障排查”等，支持快速按主题过滤。

- **多仓库聚合视图**：通过约定式的文件命名与目录结构，可同时索引来自多个 Git 仓库的文档链接，实现跨仓库的统一资源定位与引用，避免重复记录。

- **版本与状态追踪**：每条外链记录可附带文档版本号或最后更新时间，支持标识“有效”、“失效”、“待复核”、“已迁移”等状态，便于定期清理或更新。

- **纯 Markdown 驱动**：所有索引数据均以标准 Markdown 文件存储，无需数据库或特定后端服务，可直接托管在 GitHub、GitLab 等代码托管平台，支持 PR 审核与变更追踪。

- **自动化辅助工具**：项目附带简单的 Shell 脚本与 Python 辅助工具，用于检查链接可访问性、生成统计报告、校验文件格式规范性，降低人工维护成本。

## 应用场景

**个人技术笔记整合**：开发者可将多年积累的散落于本地笔记、浏览器书签、临时邮件中的技术外链统一录入 Olive Link Navigator 的索引体系中，按项目或技术领域分类，并随时通过全局检索快速定位。例如，在排查 Kubernetes 集群网络问题时，可直接通过索引找到之前收藏的 Calico 故障排查指南和官方流量监控面板地址。

**团队内部文档导航门户**：中小型研发团队可利用 Olive Link Navigator 搭建轻量的内部技术文档门户首页，将常用的 CI/CD 流水线配置文档、监控告警规则说明、日志查询平台地址、数据库备份恢复手册等集中在一个仓库中，通过 GitHub Pages 或静态站点生成器渲染为团队导航页，降低新成员上手的信息寻找成本。

**开源项目外部资源引用**：开源项目维护者可以在项目主仓库的 docs 目录下引入 Olive Link Navigator 的索引子模块，统一记录所有依赖的外部参考资源、上游设计文档、相关标准规范链接、社区讨论汇总等，从而让外部贡献者更清晰地理解项目的技术背景与依赖生态。

**技术调研结果归档**：在进行新技术选型或方案设计时，研究人员可将所有调研过程中查阅的官方文档、性能测试报告、社区案例、竞品分析文章等链接统一录入索引，并标记调研结论与推荐方案，形成可追溯的决策依据库，便于后续复盘或向团队汇报。

## 快速开始

以下步骤将指导您在本地环境中克隆、安装依赖并运行 Olive Link Navigator 的基础服务（静态索引预览）。

```bash
# 1. 克隆项目仓库
git clone https://github.com/zerxonhy/olive.git
cd olive

# 2. 安装依赖（需要 Node.js 18+ 和 npm）
npm install

# 3. 启动开发服务器，预览索引导航界面
npm run dev
```

完成上述步骤后，访问控制台输出的本地地址即可查看当前索引内容。如需构建静态站点用于部署，请执行 `npm run build`。

## 贡献指南

我们欢迎并感谢任何形式的贡献，无论是新增链接记录、修正文档错误、提交功能改进还是报告问题。请遵循以下步骤参与贡献：

1. **Fork 仓库并创建特性分支**：从主仓库 Fork 到您的个人账户下，然后基于 `main` 分支创建一个新的分支，分支命名建议使用 `feature/xxx` 或 `fix/xxx` 格式，以便清晰表达变更意图。

2. **遵循索引文件规范**：所有新增或修改的链接记录必须符合 `docs/guide/add-link.md` 中定义的元数据格式，包括必要字段（标题、原始链接、分类标签、状态、添加日期）。请确保 Markdown 文件通过 `markdownlint` 校验。

3. **运行本地验证脚本**：在提交 PR 之前，请执行 `npm run check` 和 `npm run test` 确保链接格式有效、无损坏引用，并且所有单元测试通过。如果新增了功能性代码，请同步补充对应的测试用例。

4. **提交 Pull Request 并描述变更**：将您的分支推送到您的 Fork 仓库后，向主仓库提交 PR。PR 描述中应清晰说明本次变更的目的、涉及的文件列表、以及任何需要特别注意的依赖更新或配置变更。

5. **参与代码审查与迭代**：项目维护者将对 PR 进行审查，可能会提出修改建议。请及时响应反馈，并在必要时更新您的分支。合并后，您的变更将自动纳入下一个版本发布。

## 常见问题

**Q: 如何批量导入已有的浏览器书签或本地 Markdown 笔记中的链接？**

A: 项目目前不提供自动导入功能，但您可以使用 `scripts/import-helper.py` 脚本（需自行根据书签导出格式调整）将 CSV 或 HTML 书签导出文件转换为符合索引格式的 Markdown 文件。对于本地笔记，建议使用正则表达式批量提取链接并手动补充元数据，以确保分类准确性。未来版本将考虑增加官方导入工具。

**Q: 索引中的链接失效时，系统如何通知维护者？**

A: 项目 CI 流水线配置了定时任务（每周一 UTC 00:00），通过 `npm run check -- --fail-fast` 检测所有已索引链接的 HTTP 状态码。如果发现状态码非 200 或发生连接超时，CI 将输出失败报告并在仓库的 Actions 页面中保留详细日志。同时，您也可以配置邮件通知或 Webhook 回调，将检测结果推送到您的团队通讯工具中。

**Q: 能否将 Olive Link Navigator 部署到公司内网环境，完全不访问外网？**

A: 完全可以。所有核心功能（索引解析、静态站点生成、格式校验、链接状态检测）均不依赖外部 API 或在线服务。您只需将 Node.js 运行时和内网可用的 npm 镜像源配置好，即可完全在内网环境中完成安装、构建和部署。唯一的限制是链接可用性检查脚本在内网环境中只能检测到内网可访问的链接，对外部公网链接会返回超时，您可以通过配置文件跳过外部链接检测或使用代理。
