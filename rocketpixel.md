# Olive Resource Navigator

Olive Resource Navigator 是一个面向技术研究者、开发者与开源贡献者的结构化外链资源聚合与导航工具。项目定位为“技术资源地图”，通过对分散在外部仓库、文档、教程与规范中的高价值链接进行人工整理、分类归档与语义标注，帮助用户快速定位特定技术主题下的关键参考材料，减少信息检索成本，提升学习与开发效率。

本项目并不试图替代搜索引擎或通用书签管理器，而是聚焦于特定技术领域（如前端工程化、跨端方案、设计系统、可视化规范等）的深度链接收集与上下文说明。每个收录的 URL 均附带项目内部的标签系统与简要注释，便于后续扩展为知识图谱或自动化巡检链路。Olive Resource Navigator 适合作为团队内部技术决策的参考资料库，也可作为个人技术博客的补充外链索引。

## 功能概览

- **按主题分类的外链目录**：所有收录链接按照技术领域、文件类型或项目阶段进行一级分类，支持快速筛选与浏览。

- **链接状态标注**：对每个外链资源标记“稳定引用”、“版本锁定”或“外部变更提醒”三类状态，辅助判断引用可靠性。

- **Markdown 原生渲染**：所有资源列表与文档导航均使用标准 Markdown 表格与代码块，可直接在 GitHub、GitLab 或本地编辑器中清晰展示。

- **批量导入与去重**：支持从外部 CSV 或 JSON 文件批量导入链接清单，自动检测重复 URL 并合并注释字段。

- **版本快照记录**：在资源列表章节中记录每次批量更新的批次号（如第 74/107 批）与日期，便于追溯外部内容的变动周期。

- **上下文关联推荐**：根据当前查看的资源文件路径，自动匹配同目录或同前缀的相关链接，形成微型阅读链路。

- **结构化目录树输出**：项目自身维护一套清晰的 ASCII 目录树，帮助新贡献者在一分钟内理解仓库组织方式。

- **扩展字段预留**：每个资源条目预留“优先级”、“所属模块”、“审核人”三个扩展字段，便于接入自动化工作流。

## 应用场景

- **技术选型前的资料梳理**：团队在引入新框架或库之前，可通过 Olive Resource Navigator 快速收集官方文档、社区最佳实践与已知问题讨论链接，形成横向对比依据。例如，在评估不同 Canvas 渲染方案时，可直接定位到相关信号处理与性能优化的外部参考。

- **开源项目维护者的依赖追踪**：当上游仓库的文档或示例代码发生变更时，维护者可通过本项目定期同步的资源列表，检查自身项目中的引用链接是否仍然有效，并批量更新快照版本。

- **技术写作与教程编撰**：撰写技术博客或内部培训材料时，作者可以利用本项目预先整理好的外链分类，快速引用权威资料，避免重复搜索。同时，链接的上下文注释可以直接转化为文内引用说明。

- **新人入职学习路径规划**：团队新人可通过本项目按主题索引的外部学习资源，从基础概念（如 Cedar 系列规范）到进阶实践（如 Coral 系列案例）逐步深入，缩短上手周期。

## 快速开始

以下命令可在本地环境中获取 Olive Resource Navigator 项目副本，并启动基础预览服务（基于 Node.js 与 Markdown 渲染工具）。

```bash
# 克隆仓库
git clone https://github.com/zerxonhy/olive-navigator.git

# 进入项目目录
cd olive-navigator

# 安装依赖（用于本地预览与链接检查）
npm install

# 运行开发服务器，访问 http://localhost:3000 查看资源导航页
npm run dev
```

## 项目结构

项目根目录采用模块化组织方式，核心资源列表存放于 `./resources/` 下，文档与脚本分离。ASCII 目录树如下所示。

```
olive-navigator/
├── resources/                         # 主资源目录，按主题分文件夹
│   ├── signal/                        # 信号处理相关外链（含 canvassignal 等）
│   │   └── index.md                   # 该分类下的链接清单与注释
│   ├── cedar/                         # Cedar 规范系列（cedarcoral, cedarwillow）
│   │   ├── spec.md                    # 规范概述与版本对照
│   │   └── references.md              # 外部参考链接汇总
│   ├── coral/                         # Coral 案例系列（coralmaple, coralsaffron）
│   │   ├── cases.md                   # 案例场景说明
│   │   └── config/                    # 相关配置示例占位
│   └── cloud/                         # 云与亮度相关资源（cloudbright）
│       └── brightness.md              # 亮度与渲染主题链接
├── docs/                              # 完整文档目录（见“文档导航”章节）
│   ├── guide/
│   ├── maintain/
│   ├── design/
│   └── extend/
├── scripts/                           # 工具脚本（链接检查、批次更新）
│   ├── check-links.js                 # 批量 HEAD 请求验证外联有效性
│   └── update-batch.sh                # 更新批次号与时间戳
├── templates/                         # Markdown 模板，用于生成新资源页
│   └── resource-template.md
├── .github/                           # GitHub 社区配置文件
│   └── workflows/                     # CI 工作流（每日链接巡检）
├── package.json                       # Node.js 项目配置
└── README.md                          # 当前文件
```

## 贡献指南

欢迎外部贡献者参与 Olive Resource Navigator 的资源扩充与文档优化。请遵循以下步骤提交变更。

1. **Fork 仓库并创建特性分支**：从主仓库的 `main` 分支 fork 到个人账户，然后新建分支，分支命名建议使用 `feat/资源主题` 或 `fix/链接更正` 格式。

2. **更新资源列表或文档**：若添加新外链，请先在 `resources/` 下找到对应主题目录，修改 `index.md` 或相关文件，并确保 URL 格式严格遵循原样复制原则，不得添加或修改协议与域名前缀。

3. **运行本地检查脚本**：在提交前，执行 `npm run check` 以运行链接可访问性检测与 Markdown 格式校验。若检查失败，请根据输出提示修正。

4. **提交 Pull Request**：在 PR 描述中清晰说明本次变更的批次、新增链接的用途以及是否影响现有分类结构。PR 至少需要一位维护者审核通过。

5. **合并后的同步**：PR 合并后，维护者将更新批次号（如 75/107）并触发 CI 重新生成目录树快照，确保项目状态一致。

## 常见问题

**Q：资源列表中的链接访问返回 404，我应该如何处理？**

A：首先确认链接是否因外部仓库分支更名或文件移动导致失效。若确认链接已变更，请在资源列表对应条目中标注“已迁移”，并在 Pull Request 中提供新的可用链接。若原链接内容已删除且无替代，则需在列表中移除该条目，并在批次更新日志中说明。

**Q：我能否将本项目用于商业产品内部文档？**

A：可以。本项目采用 MIT 许可证，允许商业使用、修改与再分发。但请注意，项目本身收录的外部链接版权归属其原始作者，本项目仅提供导航引用，不主张对外部内容的任何权利。

**Q：如何请求新增一个未在现有分类中覆盖的技术主题？**

A：请在 GitHub Issues 中提交“主题新增”类型的请求，并附上至少 3 个初始参考链接及简要理由。维护者会评估后决定是否新建主题目录，并通知您后续的贡献流程。
