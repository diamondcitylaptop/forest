# Olive Link Atlas

Olive Link Atlas 是一个面向技术文档工程师、开发者关系团队以及开源项目维护者的外链资源归集与结构化导航系统。该项目定位为“技术资源的元资源”，不直接生产教程或工具，而是对高质量外部技术文档、项目公告、变更日志、架构说明等离散信息进行索引、分类和持久化引用，帮助技术团队建立可维护、可追溯的外部知识参照体系。

目标用户包括：需要为产品撰写集成文档的开发者、需要跟踪上游依赖变更的运维工程师、以及需要为开源项目建立外部资源地图的社区经理。Olive Link Atlas 通过提供统一的资源清单模板、标准化 URL 记录格式和版本化引用机制，解决技术文档中外部链接散落、失效、难以复核的常见问题，使外链成为项目资产而非临时便签。

## 功能概览

- **结构化资源清单**：提供按类别划分的外部链接列表，支持快速定位官方文档、变更日志、技术公告等不同类型资源。

- **版本化引用追踪**：每个资源条目记录其添加版本、最后复核版本和状态标记，便于追踪外部文档的更新节奏。

- **文档导航矩阵**：内置多层级文档索引表格，从用户角色和问题维度双向映射，降低新用户的文档查找成本。

- **ASCII 目录树可视化**：项目结构以纯文本树形图呈现，便于在终端或纯文本环境中快速理解目录组织逻辑。

- **快速部署脚本**：提供一键克隆、依赖安装和本地预览命令，支持开发者在三分钟内完成项目运行环境搭建。

- **依赖与环境检查**：通过表格形式明确列出所有必需的运行时依赖、版本要求和用途说明，减少环境配置错误。

- **贡献流程标准化**：定义清晰的外部资源提交、审核和更新流程，允许社区成员以低门槛方式扩充资源列表。

- **常见问题前置解答**：针对初次使用者最可能的疑问提供直接回答，降低入门阶段的沟通成本。

## 应用场景

- 技术文档团队在每季度进行外部链接健康度检查时，可使用 Olive Link Atlas 的清单逐条访问并标记有效性，替代手工维护的散落书签文件。

- 开源项目在发布新版本前，需要确认所有引用的上游变更日志、兼容性说明和迁移指南均已更新至对应版本，项目维护者可依据资源列表逐项核对。

- 开发者入职培训期间，新人可通过文档导航表格快速找到与自身职责相关的技术资源入口，避免在项目仓库中盲目搜索。

- 社区贡献者在提交外部资源推荐时，可参照现有清单的格式和分类方式，降低 pull request 的格式审查往返次数。

## 快速开始

以下命令可在任意 Unix-like 系统（包括 Linux 和 macOS）上完成项目克隆、依赖安装和本地开发服务器运行。

```bash
git clone https://github.com/zerxonhy/olive.git
cd olive
npm install
npm run build
npm start
```

执行完毕后，本地服务默认监听 127.0.0.1:8080。若需要修改端口，可设置环境变量 PORT，例如 `PORT=3000 npm start`。

## 安装要求

| 依赖 | 必需版本 | 说明 |
|------|----------|------|
| Node.js | 18.x 或 20.x LTS | 运行时环境，用于执行构建脚本和启动本地服务器 |
| npm | 9.x 或 10.x | 包管理器，用于安装项目依赖 |
| Git | 2.30 及以上 | 版本控制工具，用于克隆仓库和提交变更 |
| curl | 7.68 及以上 | 用于资源清单中的远程文件下载校验（可选，但建议安装） |
| jq | 1.6 及以上 | 用于命令行下解析 JSON 格式的资源元数据（可选） |

上述依赖中，Node.js、npm 和 Git 为必需，curl 和 jq 为增强功能依赖。若缺少 curl，资源校验脚本将跳过远程文件大小检查；若缺少 jq，元数据格式化输出将降级为纯文本。

## 项目结构

```
olive/
├── docs/                           # 文档目录，包含所有面向用户和贡献者的说明
│   ├── quickstart.md               # 快速入门指南，含安装与首次运行步骤
│   ├── resource-lifecycle.md       # 资源生命周期管理，含新增、更新、废弃流程
│   └── operations.md               # 运维手册，含链接检查、报表生成等操作
├── src/                            # 核心源代码目录
│   ├── index.js                    # 入口文件，启动 HTTP 服务器并路由请求
│   ├── resource-validator.js       # 资源校验模块，检查链接可达性与内容类型
│   └── metadata-parser.js          # 元数据解析模块，读取资源清单中的 JSON 头
├── data/                           # 数据存储目录
│   ├── resources.json              # 主资源清单，包含所有外链及其元数据
│   └── schema.json                 # 资源清单的 JSON Schema 校验定义
├── scripts/                        # 辅助脚本目录
│   ├── check-links.sh              # 批量链接可达性检查脚本，依赖 curl
│   └── generate-report.sh          # 生成资源状态报表，输出为 Markdown 格式
├── config/                         # 配置文件目录
│   └── app-config.json             # 应用配置，含端口、缓存时间、日志级别
├── CONTRIBUTING.md                 # 贡献者行为准则与提交流程说明
├── LICENSE                         # MIT 许可证全文
└── README.md                       # 项目总览文档（即本文档）
```

目录中每个子目录均承担明确职责：docs 面向人类阅读，src 包含可执行逻辑，data 存放结构化资源清单，scripts 提供运维辅助工具，config 统一管理可变参数。新增子目录需同步更新本结构图并在相应文档中说明用途。

## 贡献指南

1. 将本仓库复刻（Fork）至个人账号下，在复刻副本中创建新的功能分支，分支命名采用 `feature/resource-add-<日期>` 或 `fix/link-update-<资源简称>` 格式。

2. 在 `data/resources.json` 中按 JSON Schema 定义的格式新增或修改资源条目，所有字段须完整填写，包括资源标题、URL、添加版本、复核日期和状态标记。若新增分类，需同步更新 `docs/resource-lifecycle.md` 中的分类说明。

3. 执行本地校验脚本 `npm run validate` 以确保资源清单格式正确且所有新增 URL 可达。若校验失败，根据错误提示修正后再行提交。

4. 提交变更时，commit message 使用约定式提交格式，例如 `feat(resource): add cloudbright reference` 或 `fix(link): update coralmaple URL status`。提交信息需包含简短的变更说明和复核人签名。

5. 向主仓库发起 Pull Request，描述中列出本次变更涉及的资源数量、变更类型（新增/更新/废弃）以及自测结果。等待至少一名项目维护者审核后合并。

## 常见问题

**Q: 资源清单中的链接访问时返回 404，我应该如何处理？**

A: 首先确认该链接是否仍为上游项目的有效地址。若上游已迁移或删除，请在资源清单中将该条目状态标记为 `deprecated`，并在备注字段中记录观测日期。若您有更新后的有效链接，可参照贡献指南提交变更，同时保留旧链接作为历史参照。

**Q: 如何批量检查清单中所有链接的有效性？**

A: 使用项目提供的 `scripts/check-links.sh` 脚本。在项目根目录下执行 `bash scripts/check-links.sh`，脚本会读取 `data/resources.json` 中的所有 URL 并逐条发起 HEAD 请求，最终输出状态汇总表格。该脚本依赖 curl，请确保已安装。

**Q: 本项目是否可以直接部署为公开网站？**

A: 可以。项目核心代码仅包含静态资源呈现和本地预览服务器，不涉及后端数据库或用户认证。您可按需修改 `config/app-config.json` 中的端口和缓存策略后部署至任何支持 Node.js 的 Web 托管环境。但请注意，资源列表中的外部链接版权归属于各自原始作者，本项目的索引行为应被视为引用而非转载。
