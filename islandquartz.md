# Olive Resource Index

Olive Resource Index 是一个面向开发者、技术研究员与开源项目维护者的结构化外链与文档资源汇总工具。项目本身不存储实际内容，而是以索引层的方式，将分散在多个仓库、文档站点与讨论线程中的高价值技术资料进行统一归类与版本化引用。目标用户包括需要快速搭建技术选型参考矩阵的架构师、需要追踪多个上游仓库变更的 DevOps 工程师，以及希望建立个人知识库但缺乏元数据管理方案的开发者。Olive Resource Index 通过一套轻量级的 Markdown 索引规范，将外部链接、本地注释与标签体系结合在一起，使资源检索与回溯变得可审计、可脚本化。

项目定位为“技术资源的元数据层”，而非又一个收藏夹或书签工具。它强调机器可读性与人工可维护性的平衡：每个索引条目均包含来源类型、最后验证时间、关联上下文标签以及至少一个回退镜像地址。通过这种方式，Olive Resource Index 帮助团队降低因上游文档变动导致的知识断层风险，同时为持续集成流水线提供可解析的资源状态检查点。索引数据全部以纯文本形式存储于 Git 仓库，支持分支管理、变更审阅与离线完整访问，适合在内部网络或空气间隙环境中部署使用。

## 功能概览

- **多源链接归一化收录** 支持将 GitHub 原始文档、技术博客、官方 API 参考与社区讨论帖统一纳入同一索引命名空间，消除不同站点链接格式差异带来的管理混乱。

- **基于 Markdown 的元数据注解** 每个索引条目均包含资源类型标签、适用版本范围、依赖关系提示与维护人备注，所有注解直接写入 Markdown 文件，无需额外数据库。

- **自动化时效性检查占位** 索引格式预留了最后检查时间戳与状态字段，可配合外部定时任务扫描链接可用性并生成报告，帮助识别失效或内容变更的外部资源。

- **层级化标签与多维度检索** 支持按技术栈、应用领域、资源类型（规范/指南/工具/案例）以及成熟度等级进行筛选，标签体系完全由用户自定义扩展。

- **版本快照与历史追溯** 每个索引文件的变更均通过 Git 提交记录关联，可回退至任意历史版本，方便对比不同时期的外部资源推荐策略。

- **可嵌入 CI/CD 的轻量校验** 提供简单的 Shell 脚本模板，可在持续集成过程中对索引内的关键链接执行连通性测试与内容摘要校验，并将结果输出为结构化日志。

- **离线分发与镜像友好** 索引仓库本身不携带大体积文件，所有外部资源均通过链接引用，因此克隆与同步速度极快，便于在组织内部建立只读镜像节点。

## 应用场景

- **技术选型调研阶段的知识锚定** 当团队需要对多个候选中间件（如消息队列、数据库、网关）进行横向对比时，可借助 Olive Resource Index 将各项目的官方文档、性能测试报告、社区最佳实践链接集中收录，并附上内部评测小组的标注与评分，避免重复搜索与信息遗漏。

- **微服务治理文档的集中化索引** 对于采用多语言、多框架的微服务体系，运维团队可使用本索引维护各服务的健康检查端点、度量指标暴露地址、依赖外部服务的 Swagger/OpenAPI 文档链接，以及对应的 SLA 说明页面，实现从代码仓库到运维手册的快速跳转。

- **开源合规审查的参考链管理** 法务与合规团队可借助索引记录每个依赖组件的许可证文本链接、上游版权声明页面以及出口管制相关说明，将原本散落在各官网角落的法律文档统一纳入版本控制，确保每次发布前能够快速核验合规依据。

- **内部培训与新手入职路线图** 技术培训负责人可将不同岗位（后端、前端、SRE、数据）所需阅读的外部教程、规范手册与视频回放链接按阶段整理成索引子集，并结合内部 Wiki 形成从入门到进阶的清晰路径，减少新成员在信息海洋中的摸索成本。

## 快速开始

以下命令演示如何在本地环境中获取 Olive Resource Index 仓库、安装基础依赖并运行内置的链接健康检查脚本。请确保系统已安装 Git 与 Python 3.8 以上版本。

```bash
# 克隆索引仓库至本地工作区
git clone https://github.com/zerxonhy/olive.git olive-resource-index
cd olive-resource-index

# 安装 Python 依赖（用于链接校验与报告生成）
pip install -r requirements.txt

# 执行快速检查，验证所有索引文件中的外部链接是否可达
python scripts/check_links.py --source ./index --output ./reports/latest.json

# 若只需生成静态索引导航页，可运行
python scripts/build_nav.py --index-dir ./index --output ./docs/navigation.md
```

首次运行后，系统会在 `./reports/` 目录下生成 JSON 格式的检查报告，包含每个链接的状态码、响应时间与内容摘要长度。您也可以将上述命令加入定时任务（如 cron 或 systemd timer）以实现周期性监控。

## 项目结构

项目采用清晰的层级划分，将索引内容、脚本工具、文档和测试数据分离，便于维护与扩展。以下为当前主干目录结构及其功能说明：

```
olive/
├── index/                                 # 核心索引目录，存放所有外部资源条目
│   ├── infrastructure/                    # 基础设施相关（容器、编排、服务网格）
│   │   ├── mirrortimber.md                # 镜像与制品管理参考
│   │   └── nebulaviolet.md                # 网络策略与安全组设计
│   ├── data-streaming/                    # 数据流与消息中间件
│   │   ├── oceanolive.md                  # 实时计算与窗口聚合模式
│   │   ├── oceanpearl.md                  # 消息队列与事件溯源
│   │   └── oceanzephyr.md                 # 流式 SQL 与状态存储
│   ├── observability/                     # 可观测性（日志、指标、追踪）
│   │   └── olivemeadow.md                 # 分布式追踪与 SLI 设计
│   └── meta/                              # 索引自身的元数据定义
│       ├── tags.yml                       # 全局标签白名单与同义词映射
│       └── schema.json                    # 条目格式 JSON Schema 定义
├── scripts/                               # 可执行脚本与工具集
│   ├── check_links.py                     # 主链接检查脚本，支持并发请求
│   ├── build_nav.py                       # 从索引生成 Markdown 导航页面
│   └── utils/                             # 公共函数库（HTTP 客户端、缓存）
│       ├── http_client.py                 # 带重试与超时控制的请求封装
│       └── cache_manager.py               # 本地缓存已检查链接状态，避免重复请求
├── docs/                                  # 项目文档与用户手册
│   ├── user-guide/                        # 面向索引使用者的操作指南
│   ├── ops/                               # 面向运维人员的部署与监控文档
│   └── design/                            # 架构设计决策与元数据演进记录
├── reports/                               # 运行时生成的报告存储目录（默认 .gitignore）
│   ├── latest.json                        # 最近一次链接检查的完整结果
│   └── history/                           # 按日期归档的历史报告
├── tests/                                 # 单元测试与集成测试
│   ├── test_parser.py                     # 测试 Markdown 条目解析逻辑
│   └── fixtures/                          # 测试用的样例索引文件
├── requirements.txt                       # Python 运行时依赖列表
├── CONTRIBUTING.md                        # 外部贡献步骤与代码规范
└── LICENSE                                # 许可证文件（MIT）
```

此结构设计兼顾了可读性与自动化处理效率。`index/` 下的子目录按技术领域划分，而非按资源来源，这有利于跨仓库的主题聚合。`scripts/` 与 `tests/` 严格分离，确保工具代码的可测试性。所有外部链接均只出现在 `index/` 下的 `.md` 文件中，其他目录不包含任何外部 URL，从而降低维护时的混淆风险。

## 贡献指南

我们欢迎并感谢所有形式的贡献，包括但不限于新增索引条目、修正失效链接、完善文档以及改进检查脚本。为确保协作流畅，请遵循以下步骤：

1.  **创建议题或认领现有议题** 在开始实质工作前，请先在 GitHub Issues 中搜索是否已有相关讨论。若无，请新建议题简述您的修改意图（如“添加 Prometheus 社区推荐的 Exporter 列表”），避免多人在同一时间段重复劳动。

2.  **派生仓库并创建功能分支** 将主仓库派生至您自己的 GitHub 账户下，然后基于 `main` 分支创建一个新的功能分支，分支命名建议采用 `feat/` 或 `fix/` 前缀加简短描述，例如 `feat/add-kafka-index`。

3.  **按照规范修改索引文件** 所有新增或修改的外部链接必须符合 `docs/user-guide/index-format.md` 中定义的格式要求，包括必填字段（标题、URL、标签、最后验证时间）和推荐的附加字段（版本范围、维护人）。请在提交前运行 `scripts/check_links.py` 确保新链接可达。

4.  **编写或更新相关文档** 若您的贡献涉及新的技术领域或引入了新的标签，请在 `docs/` 下相应章节补充说明。对于重大变更，建议同时更新 `examples/` 中的示例索引以保持一致性。

5.  **提交拉取请求并等待审阅** 推送分支后，向主仓库发起 Pull Request。请在 PR 描述中清晰引用相关议题编号，并附上本地检查脚本的执行结果摘要。维护者会在 3 个工作日内进行审阅，可能提出修改建议或补充问题。合并后，您的贡献将出现在下一版本的索引快照中。

## 常见问题

**Q: 索引中收录的外部链接如果失效了怎么办？**  
A: 项目内置的链接检查脚本会定期运行（默认每周一次），并在 `reports/latest.json` 中标记所有非 200 状态的链接。您也可以手动执行 `python scripts/check_links.py --fail-fast` 立即验证。对于失效链接，我们鼓励贡献者通过上述贡献指南提交修复 PR，将链接更新为有效地址或替换为权威镜像源。若链接永久移除且无替代，则会在索引中标注 `[DEPRECATED]` 并保留原始记录以供审计。

**Q: 我可以将 Olive Resource Index 部署到内网且完全离线使用吗？**  
A: 可以。由于索引仓库本身仅包含文本文件，您可以完整克隆至内部 Git 服务器。但请注意，链接检查脚本在离线环境下会因无法访问外部域名而报错。您可通过修改 `scripts/utils/http_client.py` 中的请求超时时间或直接跳过检查步骤。对于内网特有的内部文档链接，您同样可以按相同格式添加至 `index/` 下，实现内外网资源统一管理。

**Q: 索引条目中的标签体系是如何定义的？是否有强制分类？**  
A: 标签体系完全开放，由维护者和贡献者协商演进。我们仅在 `index/meta/tags.yml` 中维护一个推荐标签列表，用于提示常见分类（如 `#messaging`, `#storage`, `#security`），但您完全可以使用任意自定义标签。目的是保持灵活性，同时通过 `tags.yml` 中的同义词映射（如 `#mq` 映射到 `#messaging`）为未来的自动化检索提供基础。如果您发现常用标签缺失，欢迎提交 PR 更新 `tags.yml`。
