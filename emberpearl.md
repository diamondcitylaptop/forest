# Olive 技术资源导航

Olive 是一个面向开发者和技术研究人员的开源外链资源汇总与导航项目。本项目不提供具体的软件包或代码库，而是通过精心筛选和分类的技术文章、深度分析、实验笔记等外部资源，帮助用户快速定位到特定技术领域的高质量信息。其核心定位是成为技术决策与学习路径中的“信息中转站”，解决开发者面对海量信息时筛选成本高、优质内容发现难的问题。

项目目标用户包括：正在进行技术选型的架构师、需要快速了解某个库或方案实现细节的研发工程师、以及希望跟踪特定技术领域最新动态的研究人员。Olive 通过结构化的资源索引与摘要，将分散在 GitHub 仓库、技术博客、学术论文等处的信息进行有机整合，形成可检索、可追溯的知识地图。

## 功能概览

- **按技术主题分类的资源索引**：资源按技术领域（如前端渲染、数据结构、系统设计等）进行一级分类，每个分类下聚合相关外链，方便用户按需浏览。

- **原始文档直链访问**：每个资源条目均直接链接至原始 GitHub 仓库中的 Markdown 文档，确保用户获取的是作者发布的最新版本内容，避免信息过时或转述失真。

- **多维度元数据标记**：每条资源附带包括最后更新时间、文档类型（分析/教程/实验记录）、关联技术栈等元数据，辅助用户评估相关性与时效性。

- **资源状态标识**：清晰标注每条资源的维护状态（活跃/归档）、语言（中文/英文）以及适用经验层级（入门/中级/高级），提升检索效率。

- **轻量级全文检索**：基于文档标题和描述信息，提供简单的关键词匹配检索功能，帮助用户在海量资源中快速定位感兴趣的主题。

- **社区贡献工作流**：集成 GitHub Issues 和 Pull Requests 作为资源增删改的协作渠道，允许社区成员推荐新资源或更新失效链接，保持项目活力。

## 应用场景

- **技术选型前的方案调研**：当团队需要评估不同的前端渲染方案（如 Canvas vs DOM）时，可通过 Olive 快速找到相关的对比分析文档和实验数据，缩短调研周期。

- **深入学习特定实现原理**：开发者希望深入理解某个开源库（如轻量级信号库）的内部机制时，Olive 提供的源码分析笔记和架构解读文章可作为有效的学习路径指引。

- **维护个人技术知识库**：技术负责人或高级工程师可以使用 Olive 作为外部知识源，定期同步更新其团队内部的技术周报或知识库，确保团队与社区最新实践保持同步。

- **技术社区内容发现**：对于刚接触某个技术领域的新人，Olive 的分类索引提供了一个相对完整的知识版图预览，帮助其建立学习优先级和探索路径。

- **项目文档编写参考**：在编写技术方案设计文档或 API 说明时，开发者可参考 Olive 中收录的同类优秀文档写作风格与结构，提升文档质量。

## 快速开始

以下操作将指导您在本机克隆并运行 Olive 项目的静态站点版本。

```bash
# 1. 克隆项目仓库
git clone https://github.com/zerxonhy/olive.git

# 2. 进入项目目录
cd olive

# 3. 安装依赖（假设使用 Node.js 生态）
npm install

# 4. 启动开发服务器
npm run dev
```

执行完上述步骤后，可在浏览器中访问 `http://localhost:3000` 查看项目导航页面。对于仅需浏览资源的用户，可直接访问项目 GitHub Pages 站点，无需本地运行。

## 贡献指南

Olive 项目欢迎所有形式的贡献，包括但不限于推荐新资源、更新失效链接、改进分类体系、完善项目文档以及修复页面显示问题。为确保协作效率，请遵循以下基本流程：

1.  **查找或创建议题**：在进行任何实质性更改前，请先在 GitHub Issues 中查找是否已有相关讨论。若无，请新建一个议题描述您希望解决的问题或建议的改进，以避免重复劳动。

2.  **派生仓库并创建分支**：将本仓库 Fork 到您自己的 GitHub 账号下，并从 `main` 分支切出一个新的功能分支，分支名称应简明反映更改内容，例如 `add-resource-canvas-render`。

3.  **进行更改并遵循规范**：若新增资源，请严格按照 `src/data/resources.json` 中的字段定义（如标题、URL、分类、元数据）添加条目。若更新文档，请确保 Markdown 格式正确，并尽量与现有风格保持一致。

4.  **提交更改并创建 Pull Request**：提交代码时请编写清晰的提交信息。完成后，将您的功能分支推送到您的派生仓库，并向本仓库的 `main` 分支发起 Pull Request。请在 PR 描述中关联相关议题，并简要说明更改内容与测试情况。

5.  **代码审查与合并**：维护者将审查您的 PR，可能会提出一些修改意见。通过审查后，您的更改将被合并到主分支，并随下次站点构建自动部署上线。

## 常见问题

**Q: 资源列表中的文档为什么存放在同一个 GitHub 仓库中？**

A: Olive 作为一个外链汇总项目，理论上资源可以指向任何外部 URL。当前列表中的文档被集中放置于 `zerxonhy/olive` 仓库中，是为了确保在项目初期，这些分析类文档能与索引站点保持版本一致性和访问稳定性。未来，项目支持将外部独立博客或文档站点的优质文章也纳入索引。

**Q: 我发现某个资源链接失效了，应该如何报告？**

A: 您可以通过项目 GitHub 仓库的 Issues 页面提交一份“链接失效”报告。请新建一个 Issue，在标题中注明失效资源的标题或部分 URL，并在正文中提供该资源的原始链接（若还记得）以及您遇到的访问错误信息。维护者会尽快核实并更新或移除该条目。

**Q: 我想推荐一个不在当前列表中的优质技术资源，该怎么做？**

A: 非常欢迎。请按照上述“贡献指南”中的流程，新建一个议题或直接发起一个 Pull Request，在 `src/data/resources.json` 中添加您推荐的资源信息。请务必提供完整的资源标题、URL、所属分类、简要描述及适用的元数据（如技术栈、经验层级）。维护者会评估其相关性并决定是否收录。
