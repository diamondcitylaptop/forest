# Olive Reference Archive

Olive Reference Archive 是一个面向技术研究者、开源贡献者以及基础架构工程师的高质量外链与技术文档汇总系统。本项目并非传统的代码库或应用框架，而是一个精心编排的参考信息索引工程，旨在解决技术资料碎片化、关键文档难以追溯以及上下文丢失的问题。目标用户包括但不限于系统架构师、运维工程师、开源项目维护者以及需要对底层技术实现进行交叉验证的开发者。

通过结构化的索引机制和严格的版本引用策略，Olive Reference Archive 将分散在多个仓库与站点中的高价值技术论述、配置范例与架构决策记录进行归集与关联。项目本身不存储原始内容，而是提供一套稳定的引用锚点与元数据描述体系，确保用户在技术演进过程中能够快速定位到特定版本的权威说明，从而降低因文档漂移导致的认知负担与排错成本。

## 功能概览

- **结构化外链索引**：提供按技术领域、文档类型与维护状态分类的链接清单，每条条目均附带上下文标签与摘要说明，便于快速筛选。
- **多版本引用锚点**：针对同一技术主题的不同迭代版本建立平行引用记录，支持用户追溯设计决策的演变路径。
- **元数据增强标注**：每个资源条目均包含作者信息、最后更新时间、关联议题编号以及兼容性范围，提升信息的可验证性。
- **自动化格式校验**：内置链接有效性检查与协议一致性校验工具，确保索引库中的每个引用均符合预期的访问规范。
- **离线文档映射**：支持将远程参考文档映射至本地缓存结构，为无网络环境下的技术评审提供基础支持。
- **主题分类树浏览**：基于目录结构实现按功能模块、应用场景或技术栈层级的逐级下钻浏览，提升信息检索效率。
- **变更影响分析**：当索引中的上游文档发生更新时，系统提供关联条目影响范围提示，辅助维护者评估波及面。
- **快速启动模板**：提供标准化的引用记录模板与元数据样例，降低新贡献者的入门门槛。

## 应用场景

- **技术选型与方案评审**：在进行中间件选型或架构方案评审时，团队成员可通过本索引快速查阅特定版本的功能说明与性能基准数据，避免因查阅非官方或过时资料而产生决策偏差。
- **故障排查与根因分析**：当生产环境出现非预期行为时，运维人员能够借助索引中记录的历史配置参考与已知问题说明，快速比对当前设置与推荐值之间的差异，缩小排查范围。
- **开源贡献入门引导**：对于希望参与上游项目但缺乏上下文的新贡献者，索引中的设计文档链接与开发指南汇总可提供清晰的入门路径，降低社区参与门槛。
- **合规审计与版本追溯**：在需要遵循内部或外部合规要求时，审计人员可通过索引中的版本引用锚点，准确获取特定时间节点的策略说明与接口定义，确保审计证据的完整性。
- **离线技术培训材料准备**：培训组织者可利用索引中的离线映射功能，批量准备讲义所需的参考文档副本，确保培训环境不受外部网络波动影响。

## 快速开始

以下步骤指导您完成 Olive Reference Archive 项目的本地克隆、依赖安装与初始运行。

```bash
# 步骤 1: 克隆项目仓库
git clone https://github.com/zerxonhy/olive.git
cd olive

# 步骤 2: 安装依赖（项目基于 Node.js 20 LTS 与 pnpm）
corepack enable
pnpm install

# 步骤 3: 执行本地索引构建与验证
pnpm run build:index
pnpm run validate:links

# 步骤 4: 启动本地预览服务（默认端口 4173）
pnpm run preview
```

执行上述命令后，您可以通过终端输出的本地地址访问索引预览界面。若需要自定义构建输出目录或修改校验规则，请参阅 `docs/configuration.md`。

## 项目结构

项目采用模块化目录组织，各子目录职责清晰，便于维护与扩展。以下为项目核心目录树结构及其功能说明。

```
olive/
├── index/                          # 核心索引数据目录
│   ├── entries/                    # 每一条外链的元数据定义（YAML 格式）
│   │   ├── core/                   # 核心基础组件相关条目
│   │   ├── network/                # 网络协议与通信类条目
│   │   └── storage/                # 存储与持久化类条目
│   └── schemas/                    # 索引数据结构的 JSON Schema 定义
├── scripts/                        # 构建与工具脚本
│   ├── build-index.js              # 主索引构建脚本
│   ├── validate-links.js           # 链接有效性校验器
│   └── offline-mapper.py           # 离线文档映射生成工具（Python）
├── docs/                           # 项目文档（用户、贡献者、维护者指南）
│   ├── user-guide/                 # 面向终端用户的浏览与检索说明
│   ├── contributor/                # 面向贡献者的条目编辑规范
│   └── maintainer/                 # 面向维护者的发布与检查流程
├── public/                         # 静态资源（预览界面的样式与前端脚本）
│   ├── css/                        # 基础样式表与主题变量
│   └── js/                         # 前端交互逻辑（搜索过滤、树形展开）
├── cache/                          # 本地缓存目录（存放映射下载的临时文件）
│   └── remote-refs/                # 远程文档的本地快照（默认不纳入版本控制）
├── config/                         # 项目配置文件目录
│   ├── app.config.json             # 预览服务端口、日志级别等运行时配置
│   └── validation-rules.json       # 链接校验的自定义规则（超时、重试策略）
├── test/                           # 单元测试与集成测试脚本
│   ├── unit/                       # 针对索引解析器的单元测试
│   └── integration/                # 针对链接校验与构建流程的集成测试
├── .github/                        # GitHub 社区健康文件与 CI 工作流
│   └── workflows/                  # 定时链接检查与索引构建自动化流水线
├── package.json                    # Node.js 项目清单（依赖、脚本入口）
├── pnpm-workspace.yaml             # pnpm 工作区配置（如需 monorepo 扩展）
└── README.md                       # 项目入口说明文档（本文件）
```

## 贡献指南

我们欢迎并感谢任何形式的贡献，包括但不限于新增引用条目、更新失效链接、改进文档内容以及完善校验工具。请遵循以下步骤参与本项目：

1.  **查阅现有议题与文档**：在提交变更前，请先浏览 `docs/contributor/` 目录下的编辑规范，并检查 GitHub Issues 中是否存在相关讨论，避免重复工作。
2.  **派生项目并创建特性分支**：将本仓库派生至您的个人账户，并在本地基于 `main` 分支创建一个描述性的特性分支，例如 `feat/add-storage-entries`。
3.  **执行变更与本地验证**：在分支上完成条目新增或修改后，请运行 `pnpm run validate:all` 执行完整的链接校验与格式检查，确保所有检查通过。
4.  **编写清晰的提交信息**：提交变更时，请使用语义化的提交信息格式（如 `feat: add new reference for cache policy` 或 `fix: update expired URL in marblejade`），便于追溯历史。
5.  **发起拉取请求并等待评审**：将特性分支推送至您的派生仓库，随后向本仓库的 `main` 分支发起拉取请求。请求中请描述变更动机、涉及的资源链接以及验证结果。

## 常见问题

**问：索引中的链接出现访问超时或返回 404，我应该如何处理？**

答：首先请确认您当前的网络环境能够正常访问 GitHub 原始内容。若确认网络正常但链接仍然失效，请检查该链接对应的文件是否在上游仓库中被重命名、移动或删除。您可以通过该仓库的提交历史进行定位。若确认为永久失效，请按照贡献指南中的流程，提交更新或移除该条目的拉取请求。

**问：预览界面中的索引内容与远程仓库中的 markdown 文件不同步，是什么原因？**

答：Olive Reference Archive 的预览服务默认基于本地 `index/entries/` 目录下的元数据构建，而非直接读取远程文件。如果您需要更新某个远程 markdown 文件的摘要信息，必须同时修改对应的本地元数据条目，并重新执行 `pnpm run build:index` 命令。同步远程文件内容的离线映射功能请参考 `docs/user-guide/offline-sync.md`。

**问：如何在 CI 流水线中自动运行链接校验，避免合并失效引用？**

答：项目根目录下的 `.github/workflows/` 中提供了参考用的 GitHub Actions 工作流定义。您可以直接复用其中的 `validate-links.yml` 文件，将其配置为在拉取请求时自动触发。对于自托管的 CI 环境，您可以调用 `scripts/validate-links.js` 脚本并传入相应的参数，脚本会以非零状态码退出以标识校验失败。
