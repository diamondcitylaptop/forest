# Horizon Nexus

Horizon Nexus 是一个面向技术决策者、架构师与高级开发人员的结构化外链与资源聚合工具。该项目不提供代码库或运行时软件，而是以精心维护的 Markdown 资源索引为核心，将分散于 GitHub 仓库中的高价值技术文档、架构分析报告与领域深度文章进行集中归类与导航。其目标用户包括需要快速追踪特定技术主题的研发团队、进行技术选型调研的架构师，以及希望建立系统化知识图谱的资深工程师。

Horizon Nexus 解决的核心问题是信息过载与资源碎片化。在开源生态中，大量优质内容深埋于仓库的目录树中，缺乏语义化的入口与跨项目的关联索引。本项目通过固定格式的索引文件（如 `horizonwillow.md`、`islandcrystal.md` 等），将分散的技术笔记、案例研究和实验数据整合为统一的导航体系，使得用户可以通过一份文档即可定位到特定领域（如分布式系统、前端性能、云原生基础设施）的深度资源，极大缩短信息检索路径，提升知识获取效率。

## 功能概览

- **语义化资源索引**：每个索引文件对应一个独立的技术领域或子主题，通过文件名前缀（如 `jade` 系列代表云原生与容器生态）即可快速识别内容范畴。

- **多维度交叉引用**：索引内部提供跨文件的链接跳转，支持按技术栈、应用场景或作者维度进行资源聚类，方便用户从任意入口展开网状探索。

- **版本化快照追踪**：每个索引文件的提交历史记录了资源的更新与淘汰状态，用户可通过 Git 日志回溯特定时间点的推荐资源集合。

- **轻量级元数据标注**：每条外链均附带来源说明、内容摘要与适用版本范围（如适用），帮助用户在点击前评估资源的相关性与时效性。

- **本地化快速检索**：项目根目录提供 `INDEX.md` 总览文件，汇总所有子索引的核心标签，支持基于文本搜索或 grep 命令的快速定位。

- **自定义扩展钩子**：提供统一的 YAML Frontmatter 模板，便于用户添加自有资源条目，并保持与现有索引格式的一致性。

- **社区驱动更新**：通过 Pull Request 流程接收外部贡献，所有新增或修改的链接须通过格式校验和内容相关性审核。

## 应用场景

- **技术选型调研**：当团队需要评估多个开源方案（如消息队列、API 网关或日志采集器）时，可通过 `horizonwillow.md` 获取横向对比文章的汇总链接，快速获取社区真实反馈。

- **架构评审准备**：架构师在撰写系统设计文档或进行内部技术分享前，可查阅 `islandcrystal.md` 和 `jadeatlas.md` 中的案例分析与性能调优记录，作为设计决策的依据。

- **新成员技术布道**：团队负责人可将本项目作为新入职工程师的技术阅读清单，通过 `islandfield.md` 和 `jademaple.md` 中的入门与进阶资源，帮助新人系统性地建立知识基线。

- **离线知识库构建**：在无法直接访问外网的环境下，运维人员可结合本项目索引与本地镜像工具，批量下载被引用的资源，构建内部技术文档站。

## 快速开始

```bash
# 克隆项目仓库
git clone https://github.com/zerxonhy/olive.git

# 进入项目根目录
cd olive

# 安装依赖（仅用于本地格式校验与链接检测）
npm install -g markdown-link-check

# 运行链接有效性检查（建议在提交前执行）
find . -name "*.md" -exec markdown-link-check {} \;

# 启动本地预览服务（若已配置静态站点生成器）
# 例如使用 VitePress 或 Docsify，按需调整
# npm run docs:dev
```

## 贡献指南

1.  **复刻与克隆**：复刻本项目仓库至个人账号，然后克隆到本地开发环境。确保本地 Git 配置正确，并已设置上游远程分支。

2.  **创建特性分支**：从 `main` 分支切出新分支，分支命名建议遵循 `feat/add-resource-{主题}` 或 `fix/update-link-{文件名}` 格式，以清晰表达变更意图。

3.  **编辑索引文件**：根据目标主题修改对应的 `.md` 索引文件。所有新增外链须遵守 `[描述文本](URL)` 格式，并在链接后补充 `(来源: 作者/组织)` 和 `(适用版本: x.x)` 元数据。若需新增主题分类，必须同步更新 `INDEX.md` 中的标签列表。

4.  **本地校验**：运行项目根目录下的 `scripts/check-links.sh` 脚本，确保所有外链可访问且无重定向。同时检查 Markdown 语法是否正确，表格与列表缩进保持一致。

5.  **提交与推送**：提交变更时，使用规范化的提交信息，例如 `docs: add five links to jadeatlas about gateway performance`。推送分支至远程复刻仓库。

6.  **发起 Pull Request**：通过 GitHub 界面从复刻分支向主仓库 `main` 分支发起 PR。PR 描述中需列出本次新增或修改的资源条目数量及主要变更原因，等待维护者审核与合并。

## 常见问题

**Q: 索引文件中的链接失效了怎么办？**

A: 请通过 GitHub Issues 提交链接失效报告，选择对应的 `Broken Link` 模板，并附上失效链接所在的文件名与行号。维护者会定期检查并更新或移除失效条目。您也可以自行按照贡献指南提交 PR 修复，将链接替换为有效的存档地址或更新后的文章页面。

**Q: 是否可以请求添加特定主题的新索引文件？**

A: 可以。请在 Issues 中选择 `Feature Request` 模板，详细描述您希望覆盖的新主题领域、目标受众以及您已收集到的若干候选资源示例。项目维护者会根据主题的通用性和社区需求评估是否新增文件。在获得批准前，您也可以先在现有索引文件的末尾以 `## 待扩充` 章节形式临时存放相关链接。

**Q: 如何保证资源的质量和相关性？**

A: 所有索引条目均需经过至少一位维护者的人工审核。审核标准包括：资源作者或组织的技术信誉、内容是否提供了独特的见解或数据、是否与索引主题高度相关、以及发布日期是否在 18 个月以内（经典理论或标准规范除外）。长期未更新的资源会被标记为 `(历史存档)` 提示。
