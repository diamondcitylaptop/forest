# Olive Resource Catalog

Olive Resource Catalog 是一个面向开发者与技术研究人员的轻量级外链资源汇总与导航工具。项目本身不存储任何实际内容，而是以结构化方式索引并呈现分散在各类代码仓库、文档站点与在线笔记中的技术参考链接，帮助团队或个人在内部知识体系中快速定位关键资源。

Olive 的定位并非通用书签管理器或社交化收藏工具，而是一个可版本化、可审查、可部署为静态站点的资源清单维护框架。它适合那些希望用 Git 工作流管理技术外链、并能够与现有文档站点或 CI/CD 流水线无缝集成的用户。通过将资源列表以纯 Markdown 形式组织，Olive 降低了维护门槛，同时保留了完整的变更历史与协作能力。

## 功能概览

- **结构化资源索引**：支持按项目、领域、文档类型等多维度对链接进行分类，并提供统一的条目格式规范，确保所有收录资源具备可预期的字段完整性。

- **纯 Markdown 驱动**：所有资源列表与元数据均以 Markdown 文件形式存储，无需数据库或复杂后端，可直接在代码仓库中维护，并兼容主流静态站点生成器。

- **版本化协作流程**：基于 Git 分支与合并请求，允许团队成员新增、修改或弃用资源链接，变更历史可追溯，支持审阅与讨论。

- **自动化链接检查**：内置简单的链接可达性检测脚本，可在 CI 环境中定期运行，识别失效或重定向的外部资源，帮助保持列表质量。

- **自定义分类标签**：用户可为每条资源赋予一个或多个分类标签，并支持按标签进行快速筛选与统计，便于构建不同主题的视图。

- **轻量级部署输出**：项目提供基础模板，可将资源列表渲染为静态 HTML 页面，适合挂载至内部文档站点或对象存储服务，实现零运维成本访问。

## 应用场景

- **团队技术文档外链管理**：研发团队在内部 Wiki 或文档站点中频繁引用外部依赖、规范或参考实现，但分散的链接容易失效或难以同步更新。Olive 提供单一可信源，集中维护所有外链，并通过版本控制确保变更可追溯。

- **开源项目 README 资源附录维护**：开源项目维护者需要在 README 中列出大量社区资源、教程或相关工具，但直接写在 README 中会冗长且难以维护。使用 Olive 将资源清单独立管理，并通过自动生成或包含的方式整合到项目文档中。

- **技术培训与学习路径组织**：培训负责人可为不同技术方向整理学习资源清单，包括文档、视频、在线课程等，并通过分类标签区分难度等级或主题领域，学员可基于清单进行系统化学习。

- **合规与安全审查辅助**：安全团队或合规部门需要定期审查开发人员引用的外部资源是否符合组织策略。Olive 的结构化列表配合链接检查脚本，能够快速生成待审查清单，并标记出新增或变更的条目。

## 快速开始

以下步骤帮助您在本地环境中完成 Olive Resource Catalog 的初始部署与运行。

```bash
# 1. 克隆项目仓库
git clone https://github.com/your-org/olive-resource-catalog.git
cd olive-resource-catalog

# 2. 安装依赖（项目使用 Python 3 与 pip 管理工具脚本）
pip install -r requirements.txt

# 3. 运行本地预览服务（默认监听 8000 端口）
python serve.py --port 8000
```

完成上述操作后，打开浏览器访问 `http://localhost:8000` 即可查看资源列表的静态渲染页面。若只需维护 Markdown 源文件，可直接编辑 `catalog/` 目录下的对应文件，无需启动服务。

## 贡献指南

Olive Resource Catalog 欢迎外部贡献者参与资源条目补充、工具脚本改进以及文档完善。请遵循以下步骤：

1. **讨论与提名**：在提交变更之前，建议先在项目 Issue 列表中搜索是否已有相关讨论。若无，可新建 Issue 说明拟新增的资源链接或拟改进的功能，并附上简要理由。

2. **派生与分支**：将本仓库派生至个人账户下，并基于 `main` 分支新建一个描述性的功能分支，例如 `feat/add-kubernetes-resources` 或 `fix/update-link-check-timeout`。

3. **实施变更与自测**：按照 `docs/user-guide.md` 中的格式规范编辑 `catalog/` 目录下的文件。若涉及脚本改动，请确保在 `tests/` 目录下补充或更新相应测试用例，并在本地运行全部测试通过。

4. **提交与合并请求**：提交变更时请使用清晰的提交信息，概括变更内容与影响范围。随后在 GitHub 上发起合并请求，并关联相关 Issue。项目维护者将在三个工作日内进行审阅。

5. **审阅与合并**：审阅过程中可能会有修改意见，请及时响应。合并后，变更将自动触发 CI 链接检查与静态预览更新，确保列表整体质量。

## 常见问题

**问：Olive 是否支持私有仓库或内网地址的资源收录？**

答：Olive 本身不限制链接协议或域名，任何 HTTP/HTTPS 链接均可收录。但链接检查脚本默认会尝试对外网资源进行可达性验证，若内网地址无法从 CI 环境访问，可在配置文件中将该地址加入跳过检查白名单。静态渲染页面完全在本地生成，不会主动访问外部链接。

**问：如果资源清单非常庞大，页面渲染性能会受影响吗？**

答：项目设计时已考虑千级条目规模。渲染脚本采用惰性读取与分页生成策略，默认每页展示 100 条资源，并支持按分类标签进行索引过滤。若条目数量超过一万，建议使用外部搜索引擎或增加静态站点生成器的分页配置，Olive 提供了相应的配置参数。

**问：链接检查脚本发现了失效链接，应该如何处理？**

答：CI 或本地运行检查脚本后会生成报告文件，列出所有状态码非 200 或超时的链接。针对失效链接，建议优先尝试访问原站点确认是否临时故障。若站点已永久迁移，请在资源条目中更新链接地址并注明变更原因；若站点已彻底关闭，则应从列表中移除该条目，并在提交说明中标注。
