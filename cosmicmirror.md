# Olive Resource Index

Olive Resource Index 是一个面向技术调研、知识管理和研发效能场景的轻量级外链资源汇总工具。项目本身不存储具体内容，而是以结构化 Markdown 索引体系，将分散在多个仓库、文档站点和协作空间中的高价值链接进行集中登记、分类标注与版本追踪。项目定位为“技术团队的链接中心”，适合需要维护内部技术图谱、外部依赖清单或项目资产登记表的开发团队使用。

项目解决了以下问题：技术文档中链接散落于 Wiki、邮件、即时通讯和代码注释中，导致检索困难、过期链接无法及时发现、新人上手缺乏统一入口。Olive Resource Index 提供一套基于 Git 的纯文本索引方案，通过规范化的 Markdown 文件描述每个资源元数据，并支持自动化校验链接可用性。

## 功能概览

- **资源登记与分类**：支持按项目、领域、文档类型等多维度标签对链接进行登记，每个链接条目包含标题、描述、责任人和更新周期。
- **Markdown 原生编辑**：所有索引文件使用标准 Markdown 格式，可直接在 GitHub/GitLab 中预览，无需额外数据库或后端服务。
- **链接健康检查**：提供基于 GitHub Actions 的定时校验脚本，自动检测登记链接是否返回 4xx/5xx 状态码，并生成检查报告。
- **索引版本追踪**：依托 Git 历史记录，完整保留每次资源增删改的变更信息，支持回溯任意时间点的资源全景图。
- **按批次管理**：支持将资源按引入批次分组（如第 19/107 批），便于跟踪资源引入节奏和评审进度。
- **扩展元数据支持**：每个链接条目可附带优先级、依赖关系、备选地址等扩展字段，适应不同团队的管理粒度要求。
- **多仓库聚合视图**：支持通过脚本将多个独立仓库中的索引文件聚合生成全局资源目录，适用于中大型组织。

## 应用场景

- **技术选型调研**：团队在评估多个中间件或云服务时，将各候选产品的官网、文档、性能测试报告、社区讨论链接统一登记至索引，方便横向对比和评审留痕。
- **内部文档体系建设**：将分散在 Confluence、Notion、语雀以及代码仓库 README 中的关键文档链接抽取到 Olive 索引中，形成统一的“文档地图”，新人入职可快速找到核心资料。
- **外部依赖清单管理**：登记项目所引用的第三方库、镜像源、API 服务端点等外部资源链接，配合健康检查功能，在依赖服务迁移或下线时提前预警。
- **项目资产登记**：针对大型项目，将设计文档、测试报告、部署拓扑图、监控面板地址等资产链接统一管理，确保项目交接时资产不遗漏。
- **周报/月报资源汇总**：定期将团队产出的周报链接、会议录屏、演示文稿等临时性资源登记归档，形成可检索的历史资源库。

## 快速开始

以下命令演示了如何克隆项目、安装基础依赖并运行本地预览服务。

```bash
# 克隆项目仓库
git clone https://github.com/zerxonhy/olive.git

# 进入项目目录
cd olive

# 安装依赖（Node.js 环境，用于本地预览和链接校验）
npm install

# 启动本地预览服务，默认监听端口 3000
npm run dev
```

执行上述命令后，可通过浏览器访问 `http://localhost:3000` 查看索引资源的渲染页面。如需仅进行链接校验，可运行：

```bash
npm run check:links
```

该命令将扫描 `index/` 目录下所有 Markdown 文件，提取其中的链接并进行可用性检测，结果输出至 `reports/link-check-$(date).log`。

## 贡献指南

我们欢迎并感谢任何形式的贡献。请遵循以下步骤参与本项目：

1.  **Fork 仓库并创建分支**：从主仓库 Fork 一份副本到您的账户下，然后基于 `main` 分支创建一个新的功能分支，分支命名建议使用 `feature/` 或 `fix/` 前缀。
2.  **遵循索引规范**：所有新增或修改的资源条目必须严格遵循 `templates/resource-template.md` 中定义的字段格式，包括标题、描述、URL、标签和责任人信息。提交前请运行 `npm run validate` 校验格式。
3.  **提交变更并编写清晰描述**：提交信息请使用英文或中文，并遵循语义化提交规范，例如 `feat: add batch 020 resources` 或 `fix: correct broken link in anchorbloom.md`。请确保提交信息简明扼要说明变更内容。
4.  **发起 Pull Request**：将您的分支推送到 Fork 仓库，然后向主仓库的 `main` 分支发起 Pull Request。请在 PR 描述中关联对应的 Issue（如有）并简要说明变更动机和测试情况。
5.  **等待代码审查与合并**：项目维护者将审查您的 PR，可能会提出修改意见。CI 流水线会自动执行链接校验和格式检查，所有检查通过后方可合并。

## 常见问题

**Q：索引文件中的链接如果失效了怎么办？**

A：项目内置了链接健康检查脚本 `npm run check:links`，建议每周运行一次。若发现失效链接，请及时更新索引文件中的 URL 或移除条目。如果链接为临时性资源，可在描述字段中标注有效期。对于已合并入 `main` 分支的失效链接，我们建议发起一个修复 PR，将状态标记为 `deprecated` 并附上替代链接。

**Q：如何快速查找某个特定主题的资源？**

A：您可以通过两种方式检索：一是使用 `index/tags/` 目录下的标签索引文件，按主题标签（如 `database`、`frontend`）快速定位；二是在项目根目录执行 `grep -r "关键词" index/` 进行全文搜索。本地预览服务也提供了基于浏览器 `Ctrl+F` 的页面内搜索功能。

**Q：批次管理的意义是什么？如何确定资源属于哪个批次？**

A：批次管理主要用于追踪资源的引入时间和评审周期。每个批次对应一个时间段内的新增资源集合，便于团队进行周期性回顾和清理。新资源默认归入当前活跃批次（如 `batch/019`）。批次切换由管理员在 `manifest.json` 中配置，通常每月或每季度切换一次。如果您不确定应归入哪个批次，请查阅 `index/batch/current` 软链接指向的目录。
