# Olive Resource Navigator

Olive Resource Navigator 是一个面向技术调研、知识工程与研发团队协作的轻量化外链资源汇总与导航系统。项目定位于为开发者、技术决策者与内容策展人提供一套结构清晰、可扩展的资源索引框架，用于统一管理分散在代码仓库、在线文档、技术博客与内部知识库中的高价值外部链接。

项目不依赖复杂的前端框架或数据库，核心设计理念是“以 Markdown 为数据层，以目录结构为分类轴，以版本控制为协同基础”。目标用户包括：需要维护团队技术雷达的架构师、负责整理学习路径的开发者、以及希望将外部知识资产纳入内部研发体系的技术运营人员。通过将原始链接转化为带注释、带标签、带上下文的资源条目，Olive Resource Navigator 帮助团队减少信息重复查找成本，提升外部技术资源的可发现性与可追溯性。

## 功能概览

- **资源条目结构化存储**：每条外部链接以独立 Markdown 文件形式记录，包含标题、来源、摘要、适用场景、维护状态等元数据字段，支持人类可读与机器可解析。

- **多维度分类索引**：支持按技术领域（如前端、后端、AI）、资源类型（如教程、工具库、规范文档）、成熟度等级（如实验性、稳定、废弃）等多维度对链接进行归类，便于按需筛选。

- **自动化目录树生成**：项目内置脚本可扫描指定资源目录，自动生成 ASCII 目录树快照，方便在 README 或文档站点中展示资源全貌。

- **链接可用性健康检查**：提供周期性（如每周）的链接可达性检测工具，自动标记失效或重定向的外部资源，帮助维护资源库的时效性。

- **版本化变更追踪**：依托 Git 进行资源增删改的版本管理，每次修改均附带提交信息，支持回溯任意历史状态的资源清单。

- **资源间关联引用**：支持在资源文件中通过特殊语法（如 `[[资源ID]]`）建立相互引用，形成知识图谱雏形，便于发现主题关联。

- **快速检索与过滤**：提供命令行接口（CLI）工具，支持按关键词、标签、更新时间范围等条件对资源条目进行快速过滤与输出。

- **模板化新增流程**：提供标准资源录入模板，引导填写必填字段，降低新人贡献门槛，保证数据一致性。

## 应用场景

**技术团队内部知识库建设**：研发团队可将分散在个人书签、即时通讯群聊中的技术文章、开源项目地址、最佳实践指南集中导入 Olive Resource Navigator，形成团队统一的技术参考资料入口，新成员入职时可快速了解技术栈相关学习资源。

**开源项目文档外部链接管理**：开源项目维护者可使用本系统管理项目文档中引用的所有外部依赖文档、规范标准、参考实现等链接，在版本发布前统一检查链接有效性，避免文档中出现死链影响用户体验。

**技术调研与选型过程记录**：进行技术选型或竞品分析时，调研人员可将收集到的所有相关资源（官方文档、性能对比报告、社区评价、替代方案）按主题归类，并附加调研结论和评分，形成可复查的决策依据库。

**技术培训课程配套资源索引**：培训机构或企业内训讲师可为每门课程维护一份外部阅读材料清单，按课程模块组织，学员可通过统一入口访问所有延伸阅读内容，避免因链接分散导致的学习中断。

## 快速开始

以下命令演示了从克隆仓库到启动本地资源导航服务的完整流程。

```bash
# 克隆仓库
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装依赖（Python 3.9+ 环境）
pip install -r requirements.txt

# 执行本地预览服务（默认监听 127.0.0.1:8000）
python serve.py --port 8000

# 或仅生成静态资源索引文件（用于 CI 集成）
python build.py --output docs/index.md
```

执行上述命令后，访问 `http://127.0.0.1:8000` 即可浏览当前资源库的 HTML 概览页面，包含分类导航、最近更新列表和全文检索入口。

## 贡献指南

1. 从本仓库 Fork 一份个人副本，并在本地创建新的功能分支（如 `feature/add-new-resource`），确保分支名称清晰反映变更内容。

2. 在 `resources/` 下选择合适的分类子目录，复制 `templates/resource_template.md` 并重命名为目标资源文件，填写所有必填元数据字段（标题、链接、摘要、标签、成熟度等），并确保链接格式正确。

3. 提交前执行本地验证命令 `python cli/validator.py --path resources/`，检查新增或修改的资源文件是否符合格式规范，并根据输出提示修正所有错误或警告。

4. 在提交信息中使用约定式提交格式（如 `feat: add atlascanvas resource entry` 或 `fix: update brightpixel link status`），便于生成清晰的变更日志。

5. 向原仓库发起 Pull Request，并在描述中简要说明资源新增或修改的原因、适用场景以及是否关联现有议题（Issue）。项目维护者将在 3 个工作日内完成审核。

## 常见问题

**Q：项目是否必须使用 Python 环境运行？能否只使用静态 Markdown 文件？**

A：Python 环境仅用于本地预览服务、链接检查工具和索引生成等辅助功能。资源数据本身全部以纯 Markdown 文件存储，您可以完全不运行任何 Python 脚本，仅通过文本编辑器或 IDE 直接管理和阅读资源条目。Python 工具提供的是增强体验，并非强制依赖。

**Q：如何处理外部链接失效或内容变更的情况？**

A：项目提供了 `check_links.py` 脚本，可定期（建议每周）运行以检测所有收录链接的 HTTP 状态码。对于返回 404、500 或超时的链接，脚本会生成失效列表报告。您可以根据报告手动更新或移除对应资源条目。对于返回 301/302 重定向的链接，建议更新为最终目标 URL，避免多次跳转影响访问速度。

**Q：资源条目中的标签和分类体系能否自定义？**

A：可以。分类目录结构直接在 `resources/` 下通过文件夹名称定义，您可以根据团队需求增删改分类。标签字段（`tags`）在资源文件的 YAML 元数据中维护，采用自由字符串列表形式，无预设枚举限制，您可根据实际内容灵活标注，只需保持团队内部命名一致性即可。
