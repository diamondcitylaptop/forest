# Olive Index

Olive Index 是一个面向开发者与技术研究者的轻量级外链资源汇总与导航工具。它不存储任何原始内容，仅通过结构化 Markdown 索引文件对分散在互联网各处的优质技术文档、项目笔记与经验总结进行集中归类与版本追踪。项目定位为“技术资源的元数据层”，帮助个人或小团队在信息过载的环境中建立自己的外部知识引用体系。

目标用户包括：需要维护项目外部依赖文档的架构师、频繁查阅第三方技术方案的后端工程师、以及希望将零散收藏夹转化为可维护资源清单的技术写作者。Olive Index 通过约定优于配置的目录规则，将外链资源按主题域划分，并以纯文本形式托管于 Git 仓库，使得资源列表天然支持版本控制、差异对比与协同更新，从而解决传统浏览器书签无法结构化、无法团队共享、无法追溯变更历史的核心痛点。

## 功能概览

- **结构化资源索引**：基于 Markdown 文件维护外部链接清单，每个条目支持标题、描述、标签与状态标记，便于快速检索与筛选。
- **自动化链接校验**：集成轻量级 CI 检查脚本，定期探测索引中每个外链的可达性，并对失效链接生成警告报告。
- **标签化分类体系**：支持为每条资源赋予多个维度标签（如语言、框架、应用场景），实现多角度聚合查看。
- **版本化变更日志**：每次对索引文件的增删改操作均通过 Git Commit 记录，配合模板自动生成资源变更摘要。
- **静态站点生成适配**：索引数据可直接被静态网站生成器（如 Hugo、VuePress）消费，一键发布为内部知识库门户。
- **按需导出子集**：提供命令行工具，支持根据标签或关键词过滤导出特定主题的资源列表，用于生成技术周报或项目依赖清单。
- **外部元数据缓存**：对每个外链资源自动抓取页面标题与摘要描述作为缓存元数据，减少源站请求频次，提升列表加载速度。
- **多人协同编辑锁**：通过 Git LFS 与分支策略，避免多人同时编辑同一索引文件时产生冲突，适用于团队内部共用资源库。

## 应用场景

**场景一：技术团队内部知识库的入口管理**  
开发团队在日常工作中会积累大量第三方库文档、运维手册链接与设计规范外部参考。Olive Index 可将这些分散链接汇总为统一索引，并结合 CI 自动检查链接有效性，确保团队成员始终能访问到正确的文档地址。

**场景二：开源项目的外部依赖说明维护**  
开源项目常需引用多个上游仓库、API 文档或社区讨论帖。通过 Olive Index 维护一份 `EXTERNAL_DEPS.md`，既能向贡献者清晰列出所有依赖资源，又能利用版本化日志追踪依赖更新，避免因外部链接变动导致项目构建或使用异常。

**场景三：个人技术写作的引用素材库**  
技术博主或文档撰写者在编写多篇系列文章时，可借助 Olive Index 集中管理所有引用的外部文章、规范标准与数据来源。写作时只需从索引中按标签筛选所需条目，即可快速获取 URL，并确保多篇文章间引用的一致性与可追溯性。

**场景四：技术调研报告的资源归档**  
在进行技术选型或竞品分析时，调研人员需收集大量对比材料。Olive Index 支持按主题建立独立索引文件，记录每个备选方案的主页、性能测试报告、社区活跃度链接等，调研结束后整个索引即为一份可交付的结构化附件。

## 快速开始

以下命令演示如何在本地获取并运行 Olive Index 的基础校验工具。

```bash
# 克隆项目仓库
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装依赖（需要 Node.js 18+ 与 npm）
npm install

# 运行索引校验命令，检查所有外链的可达性
npm run check:links
```

执行上述命令后，控制台将输出每个索引文件中记录的外链状态，包括响应码与响应时间。若存在失效链接，会在项目根目录生成 `broken-links-report.json` 供进一步处理。

## 项目结构

项目采用分层目录设计，将索引数据、工具脚本、文档与配置分离，便于维护与扩展。

```
olive/
├── main/                                 # 核心索引目录，存放所有外链资源 Markdown 文件
│   ├── falconisland.md                   # 消息中间件相关资源索引
│   ├── fieldbloom.md                     # 分布式存储与布隆过滤器资源
│   ├── fieldember.md                     # 计算引擎与任务调度资源
│   ├── gardendelta.md                    # 数据增量处理与 ETL 资源
│   ├── gardenmeadow.md                   # 监控指标与观测性资源
│   └── gardenrocket.md                   # 部署工具与发布流程资源
│
├── scripts/                              # 工具脚本目录
│   ├── link-checker.js                   # 外链可达性校验主脚本
│   ├── cache-updater.js                  # 元数据缓存刷新脚本
│   └── export-filter.js                  # 按标签/关键词导出子集脚本
│
├── docs/                                 # 项目文档目录
│   ├── getting-started.md                # 快速入门指南
│   ├── index-spec.md                     # 索引文件格式规范
│   ├── ci-setup.md                       # 持续集成配置说明
│   └── cli-commands.md                   # 命令行工具完整参考
│
├── config/                               # 配置文件目录
│   ├── tags-mapping.json                 # 标签别名与分组映射配置
│   └── check-policy.json                 # 链接校验超时、重试策略配置
│
├── cache/                                # 缓存目录（自动生成，不纳入版本控制）
│   └── metadata/                         # 存放每个外链的标题与摘要缓存
│
├── reports/                              # 报告输出目录（自动生成）
```

## 贡献指南

我们欢迎并感谢任何形式的贡献，包括但不限于新增资源索引、改进校验脚本、完善文档或报告问题。请遵循以下步骤参与协作。

1.  **Fork 主仓库并创建功能分支**：从主仓库 `https://github.com/zerxonhy/olive` Fork 至个人账户，然后基于 `main` 分支创建以 `feature/` 或 `fix/` 为前缀的新分支。
2.  **遵循索引规范更新资源**：若新增或修改索引文件，请严格参照 `docs/index-spec.md` 中的字段定义与格式要求，确保 Markdown 结构合法且包含必要的元数据头部。
3.  **本地运行校验与测试**：在提交前执行 `npm run check:links` 确保所有外链可达，并执行 `npm run test`（如有）以通过基础单元测试。
4.  **提交变更并推送**：提交信息请采用约定式提交格式（如 `feat: add new resource for message queue` 或 `fix: update broken link in falconisland.md`），然后推送至个人分支。
5.  **发起 Pull Request**：向主仓库的 `main` 分支发起 PR，并在描述中清晰说明变更内容、动机以及是否影响现有功能。等待维护者审阅，审阅通过后即合并。

## 常见问题

**Q：索引文件中的外链地址发生变化，我该如何更新？**  
A：直接编辑对应的 `.md` 文件，修改其中的 URL 字段内容，然后提交变更。提交后 CI 会自动重新校验该链接。若原链接已永久迁移，建议同时更新摘要描述中的说明信息。

**Q：校验脚本报告链接超时，但浏览器中可以正常访问，怎么办？**  
A：这通常是由于源站对非浏览器请求有访问限制或响应较慢。您可以在 `config/check-policy.json` 中调整超时时间（如从 5000ms 增至 10000ms）或增加重试次数。若问题持续，可在报告中将该链接标记为“需人工确认”并忽略本次告警。

**Q：能否将索引数据用于生成静态网站？**  
A：可以。项目已预留了与静态站点生成器对接的接口。您可编写一个简单的转换脚本，读取所有 `main/*.md` 文件并提取元数据与 URL，生成对应的 HTML 页面或 JSON 数据文件，供 Hugo、VuePress 或 Astro 等框架消费。
